Gemini-Modelle der KI-Bewertungen wechseln
===========================================

Ziel
----
Die produktiven KI-Bewertungen laufen ueber diese Supabase Edge Functions:

supabase/functions/recall-evaluate/index.ts
supabase/functions/feynman-evaluate/index.ts

Wenn du dort die Modellliste aenderst, wirkt das im Projekt erst nach einem
Deploy der jeweiligen Edge Function.


1. Modellliste in den Edge Functions aendern
---------------------------------------------
Oeffne bei Bedarf eine oder beide Dateien:

supabase/functions/recall-evaluate/index.ts
supabase/functions/feynman-evaluate/index.ts

Passe oben die geordnete Modellliste an:

const MODELS = [
  "gemini-3.1-flash-lite",
  "gemini-3.5-flash",
] as const;

Das erste Modell ist das Standardmodell. Jedes weitere Modell wird der Reihe
nach als Fallback versucht, wenn das vorherige Modell mit 503/Servicefehler
scheitert. Weitere Fallbacks einfach als weitere Zeilen in MODELS ergaenzen.

Wichtig: 429/Quota-Fehler duerfen nicht auf das Fallback-Modell weitergereicht
werden, sonst kann ein einziger Recall-Klick mehrere Modellquoten belasten.
Die Schleife in der Function sollte daher nur bei Status 503 weiterlaufen:

if (status === 503) {
  continue;
}
break;


2. Lokal kurz pruefen
---------------------
Im Terminal im Repo-Wurzelverzeichnis ausfuehren:

Select-String -Path "supabase\functions\recall-evaluate\index.ts","supabase\functions\feynman-evaluate\index.ts" -Pattern "MODELS|gemini-"

Erwartung: Es erscheinen nur die gewuenschten Modellnamen.

Hinweis: VS Code kann in dieser Datei lokal "Cannot find name 'Deno'" anzeigen.
Das ist fuer diesen Deploy kein Blocker, solange die Function selbst syntaktisch
korrekt ist. Die Supabase Edge Runtime stellt Deno bereit.


3. Edge Function deployen
-------------------------
Im Terminal im Repo-Wurzelverzeichnis ausfuehren:

& "C:\Users\fue\bin\supabase.exe" functions deploy recall-evaluate --project-ref ysqpmtreljfdwtlvjzis
& "C:\Users\fue\bin\supabase.exe" functions deploy feynman-evaluate --project-ref ysqpmtreljfdwtlvjzis

Die Warnung "Docker is not running" ist bei diesem Remote-Deploy normal und kein
Problem, solange danach etwa Folgendes erscheint:

Deployed Functions on project ysqpmtreljfdwtlvjzis: recall-evaluate
Deployed Functions on project ysqpmtreljfdwtlvjzis: feynman-evaluate


4. Danach testen
----------------
Im Projekt eine Recall-Karte oder eine Feynman-Uebung oeffnen, hart neu laden
und die Bewertung starten.

Wenn die Bewertung klappt, nutzt die jeweilige produktive Edge Function die
neuen Modelle. Bei einem Fehler die Supabase Function Logs und die Google-AI-
Studio-Quota pruefen.


5. Wichtige Stolperfallen
-------------------------
- Die Datei recall_batch_tester.html ist nur ein Standalone-Test und ruft Gemini
  direkt aus dem Browser auf. Modellwechsel dort wirken nicht auf das Projekt.
- Ein Jekyll-Build ist fuer reine Modellwechsel in der Edge Function nicht noetig.
  Nur bei Frontend- oder Layout-Aenderungen neu bauen.
- Den Gemini-API-Key niemals in Dateien schreiben oder committen. Der Key liegt
  als Supabase Secret GEMINI_API_KEY.
- Wenn du nur den API-Key wechselst, musst du nicht die Function-Datei aendern.
  Dann reicht ein neues Supabase Secret und ggf. ein kurzer Test.
- Wenn Google 429 meldet, ist das ein Quota-/Rate-Limit. Dann nicht automatisch
  auf ein anderes Modell ausweichen, ausser du willst bewusst dessen Quote nutzen.


6. Rollback
-----------
Wenn ein neues Modell im Projekt nicht funktioniert:

1. MODELS wieder auf die vorherige funktionierende Reihenfolge setzen.
2. Den jeweiligen Deploy-Befehl aus Schritt 3 erneut ausfuehren.
3. Die betroffene Bewertung im Projekt erneut testen.
